# CONTEXT.md — agent_RAG_1_ly 领域术语

## 领域定位

本项目的目标是构建一个覆盖「饥荒联机版（DST）Mod 开发」全部必要知识的 RAG 问答 Agent。

## 知识库四分法

知识按来源与查询模式分为四类，每类为一个独立的知识源（ChromaDB collection）：

### 本体代码（game-scripts）
- 来源：DST 游戏安装目录 scripts.zip 解压出的 Lua 脚本
- 特点：全英文、注释极少、量极大（~4000 文件）、权威但零散
- 典型查询：「这个函数怎么实现的」「这个组件有哪些方法」

### 优秀 Mod（reference-mods）
- 来源：社区优质 Mod 的全部代码（如棱镜）
- 特点：注释相对丰富、部分功能优于本体、部分功能本体完全缺失
- 典型查询：「要实现类似功能，好的做法是什么」

### 本地文档（local-docs）
- 来源：自编的 mod 制作教程、犯错经验记录、功能摘抄记录、设计决策文档
- 特点：中文撰写、最全面（含测试案例和试错经验）、量最少
- 典型查询：「怎么做 X」「做 X 有什么坑」

### 词典速查（glossary-lookup）
- 来源：prefablist.lua、tuning.lua、chinese_s.po
- 特点：精确匹配型数据，英文 ID ↔ 中文名 ↔ 参数值的对照表
- 典型查询：「ice_box 中文叫什么」「AXE_DAMAGE 值是多少」

## Agent 检索策略：两阶段先词典后知识库

用户问题中常出现中文名、俗称或不精确的术语（"冰箱""熊罐"），
LLM 不应自行猜测其对应的英文 ID 或参数名——猜错会导致本体/Mod/文档
检索命中率大幅下降。

### 阶段一：词典解析

Agent 首先调用 glossary-lookup，将用户问题中的专有名词映射为精确标识符：

- 中文名 → Prefab ID（"冰箱" → `ice_box`）
- 俗称 → 正式英文 ID（"熊罐" → `beargerfur_sack`）
- 参数名确认（"斧子伤害" → `TUNING.AXE_DAMAGE`）

### 阶段二：问题重编后检索

用阶段一获得的精确标识符重编用户问题，再分路检索本体代码、
优秀 Mod、本地文档。这样每一路检索的查询词都从"猜"变成了"确定"。

### 词典缺失时的降级策略（"软化的 B"）

当 glossary-lookup 对某个术语返回空：

1. **告知用户**：「词典中未找到 "xxx"，已自动在所有知识源中检索」
2. **不阻塞流程**：Agent 直接并行检索本体、优秀 Mod、本地文档三个库
3. **标注不确定度**：回复中注明「以下结果未经词典校译，可能不够精确」
4. **仅在全局无结果时**：提示用户手动指定知识源或提供更多信息

## RAG 查询增强技术选型

"先完善补充问题再查询"与学界中多项查询重写技术相呼应。以下是与本项目最相关的四种，
按实现复杂度递增排列：

### 1. Query Expansion（查询扩展）— 最轻量

将词典返回的精确标识符直接"拼入"原查询字符串，然后检索：

```
原文：便携衣柜怎么做
词典命中：portable_wardrobe_ly
重写后：portable_wardrobe_ly 便携衣柜怎么做 制作容器服装背包
```

**本项目映射**：这正是阶段一→阶段二的流程。代价为零（词典已在检索链内），
收益是检索命中率大幅提升。

### 2. Step-Back Prompting（回退提问）— 适配跨库交叉查询

当原始问题过于具体（如"creepyeyes.lua 第 234 行的 OnUpdate 做了什么"），
向量检索可能因匹配面过窄而失败。Step-Back 将问题抽象化后再检索：

```
原始：creepyeyes.lua 第 234 行的 OnUpdate 做了什么
回退：creepyeyes entity update logic and proximity detection mechanism
```

一次额外 LLM 调用，对嵌入空间模糊查询的召回率有 5-15pp 提升。

**本项目映射**：当词典命中的查询词过于精确（单个函数名、行号）且初检结果为空时，
可触发生成回退问题后重新检索。

### 3. HyDE（假设文档嵌入）— 适配"不知道自己要什么"的模糊问题

用户问"怎么让物品不被腐烂"——词典能查到 perishable 组件和调鲜相关参数，
但无法定位"开摆 mod 的永鲜实现方式"和"调鲜组件的工作机制"这一类需要跨概念
理解的问题。HyDE 让 LLM 先凭空写一段假设答案（可能有错，无妨），
用这段假答案的向量去搜真实文档。好处是假答案和真答案在语义空间中更接近。

**本项目映射**：词典解析完成、问题重编完成后，第一次检索返回的文档数 < 阈值时，
用 HyDE 生成假设答案当作查询向量重新检索。

### 4. Query Decomposition（子问题分解，Plan-and-Execute）— 适配复杂多跳问题

用户问"棱镜的随身容器是怎么做的？我如果要仿制但改用衣柜容器而不是背包，
需要改哪些地方？"——这包含三个子问题：

1. 棱镜月轮宝盘的核心实现逻辑是什么
2. 你的便携衣柜容器代码结构是什么
3. 把前两个知识交叉比对，找出需要修改的位置

子问题可能分属不同知识源（问题1在 reference-mods，问题2在 local-docs），
且存在依赖关系（必须先答1+2，才能答3）。

**本项目映射**：未来扩展。现阶段先做词典重写，待 Agent 行为稳定后再引入
Planner 角色负责拆解多跳问题。
