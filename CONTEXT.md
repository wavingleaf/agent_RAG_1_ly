# CONTEXT.md — agent_RAG_1_ly 领域术语

## 领域定位

本项目的目标是构建一个覆盖「饥荒联机版（DST）Mod 开发」全部必要知识的 RAG 问答 Agent。

## 知识库四分法

知识按来源与查询模式分为四类，每类为一个独立的知识源（ChromaDB collection）：

### 本体代码（game-scripts）
- 来源：DST 游戏安装目录 scripts.zip 解压出的 Lua 脚本
- 特点：全英文、注释极少、量极大（~4000 文件）、权威但零散
- 典型查询：「这个函数怎么实现的」「这个组件有哪些方法」

### 优秀 Mod（reference-mods）
- 来源：社区优质 Mod 的全部代码（如棱镜）
- 特点：注释相对丰富、部分功能优于本体、部分功能本体完全缺失
- 典型查询：「要实现类似功能，好的做法是什么」

### 本地文档（local-docs）
- 来源：自编的 mod 制作教程、犯错经验记录、功能摘抄记录、设计决策文档
- 特点：中文撰写、最全面（含测试案例和试错经验）、量最少
- 典型查询：「怎么做 X」「做 X 有什么坑」

### 词典速查（glossary-lookup）

> **设计归类**：glossary-lookup 本质是**工具**（精确键值匹配），而非知识库（向量语义检索）。与 game-scripts / reference-mods / local-docs 不同，它不依赖 embedding 相似度，而是 ID↔中文名↔参数值的确定性查表。近期按知识库方式（向量匹配存入 ChromaDB）先跑通链路；远期改为 grep 精确字符串匹配。

- 来源：prefablist.lua、tuning.lua、chinese_s.po
- 特点：精确匹配型数据，英文 ID ↔ 中文名 ↔ 参数值的对照表
- 典型查询：「ice_box 中文叫什么」「AXE_DAMAGE 值是多少」

## Agent 检索策略：两阶段先词典后知识库

用户问题中常出现中文名、俗称或不精确的术语（"冰箱""熊罐"），
LLM 不应自行猜测其对应的英文 ID 或参数名——猜错会导致本体/Mod/文档
检索命中率大幅下降。

### 阶段一：词典解析

Agent 首先调用 glossary-lookup，将用户问题中的专有名词映射为精确标识符：

- 中文名 → Prefab ID（"冰箱" → `ice_box`）
- 俗称 → 正式英文 ID（"熊罐" → `beargerfur_sack`）
- 参数名确认（"斧子伤害" → `TUNING.AXE_DAMAGE`）

词典只负责返回原始映射（多个候选时全部返回），不做判断。

### 阶段二：LLM 理解上下文后智能融合，重编问题再检索

阶段一返回的是候选映射列表，不一定是最终答案。LLM 在此阶段承担关键角色：

1. **理解用户问题的上下文**——同一个中文词在不同语境下可能映射到不同 ID
2. **从候选列表中挑选最相关的标识符**——词典返回多个候选时，LLM 根据语义判断该用哪个
3. **将选中的标识符自然融入重写后的查询**——不是机械字符串替换，而是生成一个语义通顺、利于检索的新查询

不是"词典替代 LLM"，而是"词典提供候选，LLM 做判断"的协同关系。
这样每一路检索的查询词都从"猜"变成了"确定"。

### 词典缺失时的降级策略（"软化的 B"）

当 glossary-lookup 对某个术语返回空：

1. **告知用户**：「词典中未找到 "xxx"，已自动在所有知识源中检索」
2. **不阻塞流程**：Agent 直接并行检索本体、优秀 Mod、本地文档三个库
3. **标注不确定度**：回复中注明「以下结果未经词典校译，可能不够精确」
4. **仅在全局无结果时**：提示用户手动指定知识源或提供更多信息
