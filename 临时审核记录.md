# 设计讨论临时记录

> 对话日期：2026-06-29
> 来源：/grill-with-docs 核对

## 1. glossary-lookup 分类修正

**问题**：CONTEXT.md 将 glossary-lookup 与 game-scripts / reference-mods / local-docs 并列为"知识库四分法"。但前三者是向量语义检索，glossary-lookup 是精确键值匹配（ID↔中文名↔参数值），查询模式完全不同。

**结论**：glossary-lookup 应归类为**工具**（与 search_knowledge_base 平级的 lookup_glossary），而非知识库。

**规划**：
- 近期：沿用当前 RAG 方案（向量匹配存入 ChromaDB），先跑通链路
- 远期：验证通过后考虑改为 grep 精确字符串匹配（本质是键值查找，向量检索在此场景下不是最优解）

## 2. 词典 + LLM 协同，不是"词典替代 LLM"

CONTEXT.md 的阶段一→阶段二写成了"词典映射→用精确标识符重编问题"，没有明确 LLM 在重编阶段的角色。实际意图是：

- 词典只负责返回原始映射（"冰箱" → `ice_box`，多个候选时全部返回）
- LLM 负责理解用户问题上下文后，智能地挑选/融入这些映射来重写查询
- 不是机械的字符串替换，词典提供候选，LLM 做判断

## 3. 四类查询增强的分层关系

CONTEXT.md 和创新点清单将四种技术（Query Expansion / Step-Back / HyDE / Query Decomposition）并列描述，但它们的层级不同：

- **基础链路（始终开启）**：Query Expansion（词典拼入重写）——这是必要的必经管道，不是可选策略。它对应阶段一→阶段二。
- **上层策略（条件触发）**：Step-Back Prompting / HyDE / Query Decomposition —— 从参考项目继承的三策略路由，挂在基础链路之上，基础链路产出不够时启动。

## 4. 评分门控（grade_documents）的可靠性——待优化

当前 Phase 2 的 `grade_documents` 和 `generate_answer` 使用同一个 DeepSeek 模型做 self-grade：
- "no" → 重写查询 → 但重写也可能跑偏
- "yes" → 但可能只是关键词匹配，未真正理解文档内容

待优化方向：
- 评估 grade 节点的误判率（yes/no 准确度）
- 考虑评分门控的阈值设计（是否需要更细粒度的 1-5 分而非二元 yes/no）

## 5. 四分库设计目前基于推断，需实测验证

当前"单库够用但不够好"的判断是基于单库内混合了 4 个 Mod 代码的情况，并非基于"四个来源混入同一 collection"的实测。四分库的设计更多是"来源性质不同"的理论推断。

待办：Phase 4 阶段需做**混合库 vs 分库**的检索质量对比测试，用数据验证分库的边际收益。

## 8. "CRAG"命名修正

当前创新点清单 #7 将 `retrieve → grade → rewrite → re-retrieve → answer` 称为"CRAG（Corrective RAG, 2024）模式"，但实际只实现了 CRAG 的部分链路。CRAG 论文完整方案还包括：
- Knowledge Refinement（从部分相关文档中提取可用知识条再拼接）
- Web search fallback（全不相关时降级到外部搜索）

本项目当前只有 `grade + rewrite + re-retrieve`（带评分门控的单次查询重写检索），是 CRAG 的子集而非完整实现。

**结论**：
- 近期：将命名从"CRAG"改为更准确的描述（如"带评分门控的查询重写检索"），避免挂羊头卖狗肉
- 远期：CRAG 可作为 LangGraph 图复杂度演进的参考方向（Knowledge Refinement / 外部搜索降级）

## 9. 词典需要第四张表——社区术语映射表

当前 glossary-lookup 设计了三张表：prefablist.lua（ID）、tuning.lua（参数值）、chinese_s.po（中英翻译）。但这三张表都来自官方数据，无法覆盖：

- Mod 完整名 ↔ 缩写映射（"Beneath the World Below" → "BWB" → "深埋之下"）
- 玩家社区俗称/黑话（非官方术语）

**需要第四张表**：社区术语映射表，由用户反馈持续补充。知识库（尤其是词典库）应支持定期更新，用户反馈的缩写、社区术语等即为更新素材。

## 6. 复审问题断点位置——两种方案待对比

当前设计是"检索前"确认：词典重组 → 弹出卡片 → 用户确认/超时自动通过 → 检索。但 30 秒超时意味着用户切窗口后大概率错过。

另一种方案是"回答后"纠正：先检索+回答 → 回答末尾标注"以上基于 X、Y、Z 的词典映射，若有误请点击修正"。用户看到结果后再决定是否纠正。

处理到复审问题时两种方案都尝试，试错成本很低。

## 7. 词典术语解析层与"实体链接"的关系

创新点清单将词典术语解析层归为"领域创新"。通用 NLP 中存在类似模式——实体链接（Entity Linking），即模糊实体 → 规范标识符映射。但查到做实体链接的 RAG 项目不多。

二者相似但词典层的具体实现（prefablist.lua / tuning.lua / chinese_s.po 三表 + LangGraph 图节点集成）是领域特有的。暂且保留为创新点。

## 10. 检索结果多样性——MMR 待评估

当前 `similarity_search` 不做多样性控制，Top-3 可能来自同一文件的不同行号偏移，chunk 内容高度重叠。踩坑记录 #03 的事后"同文件提示"只是标注而非预防。

优化候选：用 MMR（Maximal Marginal Relevance）替换普通的 similarity_search，增加 Top-K 结果多样性。需评估 k=3 的小候选集上 MMR 是否仍有意义。

## 11. embedding 查询前缀应从硬编码移到 config.json

当前 `add_query_prefix()` 硬编码了 bge-m3 的前缀字符串，与 `config.json` 的 `model_name` 是两个独立配置点。若未来换模型，前缀不会自动跟随——要么继续加多余的前缀，要么人工同步改两个位置。

方案：在 `config.json` 的 embedding 配置段新增可选字段 `query_prefix`，`add_query_prefix()` 从 config 读取。新用户调 config 改模型名时前缀自动跟随，无需额外知晓前缀机制。

## 12. 分块质量提升 + 向量+grep 混合检索——均未入待办/创新点

**两件事在 TODO.md 和创新点清单中均未提及**：

1. **分块质量提升**（近期）：chunk_size / overlap / 分隔符效果测试尚未做，Lua 代码可能被截断在不自然位置（如 `function` 和 `end` 分离）。TODO Phase 3 未列此项。

2. **向量语义匹配 + grep 精确检索混合**（远期创新点）：bge-m3 训练语料中 Lua 代码占比极低，代码语义（函数名、变量名、模块路径）在自然语言 embedding 空间中理解弱。grep 精确匹配代码标识符，向量匹配自然语言描述，两路融合。创新点清单未列此项。

## 13. `query_prefix` 设计已有结论但未落地到 config.json

当前 `config.json` 的 embedding 段只有 `provider` / `model_name` / `hf_endpoint` 三个字段，缺少 `query_prefix`。`embedding.py` 的 `add_query_prefix()` 硬编码了 bge-m3 前缀。

已在 §11 决定将 `query_prefix` 移入 config.json 作为可选字段，但 config.json 和 embedding.py 尚未同步修改。

## 14. `embedding.py` docstring 与实际实现矛盾

`embedding.py:10-11` 的 docstring 写："query 前缀 — ... LangChain 的 HuggingFaceEmbeddings 已内置此逻辑（通过 encode_kwargs 配置）"

但踩坑记录 #11 明确记录 `HuggingFaceEmbeddings` **不支持** `query_instruction` 入参，所以才在 `nodes.py` 中手动调用 `add_query_prefix()`。docstring 声称的"已内置"正是被 #11 否定的结论。需修正 docstring。

## 15. 管理面板的 System Prompt 预览与 `agent/prompt.py` 是两套独立逻辑

`管理面板.py:310-320` 硬编码了一套 prompt 生成逻辑（`preview_lines`），而真正发给 LLM 的 prompt 由 `agent/prompt.py:build_system_prompt()` 生成。两者结构不同——例如管理面板预览里写了"每次收到用户问题，**必须**先调用 search_knowledge_base 工具"，但 Phase 2 的 `prompt.py` 已去掉此语言（检索由图节点强制执行）。**管理面板展示的预览可能已与实际 prompt 不一致。** 需将预览改为直接调用 `build_system_prompt()` 而非自行拼装。

## 16. 重写查询退化检测过于严格

`nodes.py:145` 用 `if query == question` 精确字符串比较检测重写退化。但 LLM 可能只做了微小表面变化（加空格、换同义词），字符串不等但语义完全相同，导致 `retrieve_expanded` 白跑一次。改进方向：检测重写查询是否只是原问题的微小编辑（语义等价但字符串近似），而非要求精确相等。后续门控优化时一并处理。

## 17. Docker 与本地裸 Python 的向量库路径不一致

`config.json` 中 `persist_directory: "/app/chroma_data/bge-m3"` 是 Docker 容器内路径，而 `src/config.py` 默认值是 `"./chroma_db"`（本地运行路径）。如果用户先在 Docker 跑聊天端（写入 Docker volume），然后在宿主机直接跑 Python（读 `./chroma_db`），两者指向不同目录，看起来像"库被清空了"。需在文档中提醒或代码中检测运行环境自动切换。
