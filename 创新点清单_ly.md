### 四种技术详情

按实现复杂度递增排列：

**Query Expansion（查询扩展）— 最轻量**

将词典返回的精确标识符直接拼入原查询字符串，然后检索：

```
原文：便携衣柜怎么做
词典命中：portable_wardrobe_ly
重写后：portable_wardrobe_ly 便携衣柜怎么做 制作容器服装背包
```

**触发条件**：词典命中后始终触发。代价为零（词典已在检索链内）。

**Step-Back Prompting（回退提问）**

当原始问题过于具体（如"creepyeyes.lua 第 234 行的 OnUpdate 做了什么"），向量检索可能因匹配面过窄而失败。将问题抽象化后重新检索：

```
原始：creepyeyes.lua 第 234 行的 OnUpdate 做了什么
回退：creepyeyes entity update logic and proximity detection mechanism
```

一次额外 LLM 调用，对嵌入空间模糊查询的召回率有 5-15pp 提升。

**触发条件**：词典命中的查询词过于精确（单个函数名、行号）且初检结果为空时。

**HyDE（假设文档嵌入）**

用户问"怎么让物品不被腐烂"——词典能查到 perishable 组件和调鲜相关参数，但无法定位跨概念理解的问题。让 LLM 先凭空写一段假设答案（可能有错，无妨），用这段假答案的向量去搜真实文档——假答案和真答案在语义空间中更接近。

**触发条件**：词典解析完成、问题重编完成后，第一次检索返回的文档数 < 阈值时。

**Query Decomposition（子问题分解）**

用户问"棱镜的随身容器是怎么做的？我如果要仿制但改用衣柜容器而不是背包，需要改哪些地方？"→ 拆为多个子问题，可能分属不同知识源且存在依赖关系。

**触发条件**：未来扩展。现阶段先做词典重写，待 Agent 行为稳定后再引入 Planner 角色。

### 🆕 复审问题（Review Query）

**状态**：⚪ 新灵感，待设计

**在 RAG 链路中的位置**：

```
用户提问
  → 词典术语解析（中→英映射）
  → 问题重组（精确标识符拼接）
  → 🆕【复审问题】← 在这里插入人机校验断点
     ├─ 用户选"直接用" → 重组后的查询进入检索
     ├─ 用户选"微调"   → 用户手动编辑查询文字后进入检索
     └─ 用户选"回退"   → 回到词典解析前，用户重新描述需求，重跑词典→重组流程
  → 检索 + 评分门控 + 重写 + 生成回答
```

**三个选项的语义**：

| 选项 | 行为 | 适用场景 |
|------|------|----------|
| 直接用 | 信任重组结果，直接进入检索 | 词典映射准确、重组结果符合预期 |
| 微调 | 用户手动编辑重组后的查询文字 | 方向对但措辞不够好，小改即可 |
| 回退 | **回到词典解析之前**，用户重新描述需求，重新走词典→重组流程 | 词典映射完全偏离——不是查询写得不好，而是需求没被理解 |

"回退"不是"跳过重组用原问题继续检索"——原问题本身就是需要词典解析的原始输入，回退到检查点之前让用户有机会纠正自己对系统的表述，然后重新走一遍解析管道。

**为什么是创新**：CRAG/Self-RAG 的 correct/reject/rewrite 决策是 LLM 自动做出的，用户无感知。复审问题把这个决策权交还给用户——词典阶段是最容易出错的环节（俗称→标识符映射可能歧义），出错后全链路白跑。一个确认操作的 UI 卡片代价很低、收益很高。

**可能的实现形态**：一个 Chainlit `cl.Action` 按钮组：

```
系统：已将你的问题重组为 →
      "LEGION equipment armor absorb_percent TUNING.ARMOR"

[ 直接检索 ] [ 修改查询 ] [ 重新描述 ]
```

**配置开关**（`config.json`）：

```json
"query_review": {
    "enabled": true,
    "timeout_seconds": 30,
    "auto_approve_if_no_response": true
}
```

默认开启，30 秒无操作自动通过——不阻塞高频使用场景。
